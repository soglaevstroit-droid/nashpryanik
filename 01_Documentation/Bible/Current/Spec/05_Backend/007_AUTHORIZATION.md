# 007 — Authorization

## Назначение
Authorization описывает RBAC: проверку ролей и доступа к данным компании.

## Роли
- `worker` — монтажник.
- `foreman` — прораб.
- `finance` — финансист.
- `manager` — руководительская роль.

## Источник роли
Роль берется из JWT и сверяется с `users.role` при необходимости. `companyId` также берется из JWT и используется для фильтрации данных.

## Проверки доступа
Backend выполняет:
- проверку наличия JWT;
- проверку роли;
- проверку `company_id`;
- проверку владения или доступности сущности;
- проверку состояния сущности.

## Правила по ролям
Монтажник:
- управляет только своей сменой;
- берет доступную задачу;
- выполняет этапы своей активной задачи;
- загружает фото своих действий;
- видит свои монеты.

Прораб:
- видит сотрудников;
- создает задачи;
- создает этапы;
- назначает исполнителей;
- не начисляет монеты вручную;
- не создает выплаты.

Финансист:
- видит финансовую аналитику;
- видит AI-рекомендации;
- принимает решения по выплатам;
- не редактирует начисленные монеты вручную;
- не создает задачи как прораб.

Руководитель:
- использует управленческую аналитику без обхода ролевых правил;
- не получает автоматическое право менять финансы.

## Middleware
RBAC middleware принимает список допустимых ролей:

```text
requireRole("finance")
```

Для объектных проверок используется policy layer:

```text
canAccessTask(context, taskId)
canManageTask(context, taskId)
canCreatePayment(context)
```

## Ошибки
- `UNAUTHORIZED` — нет JWT или токен недействителен.
- `FORBIDDEN` — роль или company scope не позволяют действие.
- `INVALID_STATE` — роль есть, но состояние сущности запрещает действие.

## Audit
Отказы доступа для финансовых действий и подозрительные попытки могут фиксироваться в логах безопасности. Успешные значимые действия пишутся в `audit_log`.
