# 012 — Notifications

## Назначение
Документ описывает backend-логику уведомлений. Уведомления помогают пользователю увидеть событие, но не выполняют действие самостоятельно.

## Типы уведомлений
- `ai_recommendation`;
- `large_deviation`;
- `employee_no_activity`;
- `suspicious_action`;
- `payment_ready`;
- `attention_required`.

## In-app
In-app уведомления хранятся в `notifications` и возвращаются через `/api/v1/notifications`.

Статусы:
- `new`;
- `viewed`;
- `resolved`;
- `hidden`;
- `error`.

## Push
Push может использоваться для событий внимания. Push не является источником истины: состояние хранится в `notifications`.

## Email
Email допускается как дополнительный канал, если он включен конфигурацией. Email не должен содержать секреты, токены или полные финансовые данные без защиты.

## Логика доставки
1. Доменное событие создает notification payload.
2. `NotificationService` сохраняет in-app запись.
3. Background job доставляет push или email.
4. Результат доставки логируется.
5. Пользователь открывает уведомление и backend меняет статус.

## Связанные сущности
Уведомление может ссылаться на:
- AI-рекомендацию;
- сотрудника;
- объект;
- выплату;
- отчет;
- задачу.

Доступ к связанной сущности проверяет backend.

## Ограничения
- Уведомление не создает выплату.
- Уведомление не принимает AI-рекомендацию.
- Уведомление не меняет монеты.
- Скрытие уведомления не удаляет доменное событие.
