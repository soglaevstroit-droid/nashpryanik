# 017 — Performance

## Назначение
Документ описывает производительность backend: кэширование, оптимизацию запросов, пулы соединений и работу с тяжелыми операциями.

## Пулы соединений
PostgreSQL используется через connection pool. Размер пула зависит от окружения и числа backend-инстансов.

Правило: суммарное число соединений всех инстансов не должно превышать лимит PostgreSQL.

## Оптимизация запросов
- Фильтровать по `company_id`.
- Использовать индексы Database Bible.
- Не отдавать длинную историю на главные экраны.
- Для списков использовать pagination.
- Для nullable FK использовать partial indexes.

## Кэширование
Допустимо кэшировать:
- справочники объектов;
- настройки;
- агрегаты аналитики;
- read-only отчеты.

Нельзя кэшировать так, чтобы пользователь видел чужую компанию или устаревшее финансовое решение как актуальное.

## Тяжелые операции
В background jobs выносятся:
- AI-анализ;
- экспорт отчетов;
- пересчет агрегатов;
- очистка временных файлов.

## Файлы
Фото не проходят через память backend целиком, если можно использовать streaming upload. Thumbnail и compression выполняются асинхронно, если это тяжелая операция.

## Метрики
Измеряются:
- latency endpoint-ов;
- duration DB queries;
- upload duration;
- job duration;
- error rate;
- rate limit hits.

## Целевые правила
Главные экраны ролей должны возвращать только нужные данные: монтажнику работу, прорабу сотрудников и задачи, финансисту цифры и аналитику.
