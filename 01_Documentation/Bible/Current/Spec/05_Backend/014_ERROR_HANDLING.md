# 014 — Error Handling

## Назначение
Backend должен возвращать ошибки в едином формате API Bible и не оставлять сущности в неконсистентном состоянии.

## Единый формат

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Некорректные данные запроса.",
    "details": {
      "field": "periodFrom"
    },
    "requestId": "req_001"
  }
}
```

## Классы ошибок
- `ValidationError`;
- `UnauthorizedError`;
- `ForbiddenError`;
- `NotFoundError`;
- `ConflictError`;
- `InvalidStateError`;
- `UploadError`;
- `AIError`;
- `InternalError`.

## Маппинг HTTP
- `400` — validation или invalid state.
- `401` — нет или недействительный JWT.
- `403` — роль или company scope запрещают действие.
- `404` — сущность не найдена.
- `409` — конфликт состояния или повторная выплата.
- `413` — файл слишком большой.
- `415` — неподдерживаемый тип файла.
- `429` — rate limit.
- `500` — непредвиденная ошибка.

## Транзакции
Если ошибка произошла внутри транзакции:
- транзакция откатывается;
- доменное состояние не считается измененным;
- audit не должен фиксировать успешное действие;
- технический log фиксирует ошибку.

## Пользовательские сообщения
Сообщение должно быть понятным и не раскрывать внутренние SQL, stack trace, secrets или структуру storage.
