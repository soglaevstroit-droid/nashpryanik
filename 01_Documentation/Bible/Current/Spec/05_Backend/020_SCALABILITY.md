# 020 — Scalability

## Назначение
Документ описывает масштабирование backend на большое количество пользователей, объектов, фотографий, горизонтальные инстансы, очереди и возможное выделение микросервисов в будущем.

## Большое количество пользователей
Backend масштабируется за счет:
- stateless HTTP-инстансов;
- JWT без server session для access token;
- connection pool;
- pagination;
- индексов по `company_id`, роли, статусу и периоду.

## Множество объектов
Все запросы используют `objectId` как фильтр там, где это применимо. Главные экраны не должны загружать данные всех объектов без необходимости.

## Миллионы фотографий
Файлы хранятся во внешнем storage. Backend хранит metadata в PostgreSQL и использует streaming upload. Thumbnail и compression выполняются job-ами.

## Горизонтальное масштабирование
HTTP-инстансы должны быть stateless:
- request context создается из JWT;
- файлы не хранятся локально постоянно;
- background jobs используют очередь или блокировки;
- logs централизуются.

## Очереди
Очереди используются для:
- AI-анализа;
- отправки уведомлений;
- экспорта отчетов;
- обработки фото;
- cleanup.

Job должен быть идемпотентным и безопасным к повторному запуску.

## Микросервисы в будущем
Текущая архитектура может быть модульным монолитом. Выделение микросервисов допустимо позже для:
- AI;
- notifications;
- file processing;
- analytics/reporting.

Выделение сервиса не должно менять бизнес-логику и роли.

## Масштабирование PostgreSQL
Backend должен использовать индексы и агрегаты из Database Bible. Тяжелая аналитика не выполняется на клиенте.

## Критерии готовности
Backend готов к росту, если:
- все endpoint-ы фильтруются по company scope;
- сервисы не завязаны на один инстанс;
- storage внешний;
- background jobs идемпотентны;
- финансовые операции транзакционны;
- AI не принимает финальные решения.
