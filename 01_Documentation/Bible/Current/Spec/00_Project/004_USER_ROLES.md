# 004 — User Roles

## Назначение
Документ описывает роли системы, их обязанности, права, ограничения, доступные экраны и основные действия. Ролевые правила обязательны для UI, API, базы данных, навигации и тестирования.

## Роль: Монтажник

### Обязанности
- Начинать и завершать рабочую смену.
- Брать доступные задачи объекта.
- Выполнять этапы задачи в установленной логике.
- Подтверждать ключевые действия фото.
- Сообщать, если задачу невозможно выполнить.
- Следить за текущим статусом работы.

### Права
- Видеть общий список задач объекта.
- Взять свободную задачу.
- Просматривать задачу, если она уже в работе у другого сотрудника.
- Выполнять этапы своей текущей задачи.
- Отправлять фото подтверждения.
- Видеть свой баланс монет.
- Видеть свою историю задач и профиль.

### Ограничения
- Одновременно может быть только одна задача в работе.
- Если задача назначена другому сотруднику, монтажник не может взять ее.
- Если задача уже в работе, другие задачи приглушены и доступны только для просмотра.
- Монтажник не видит тариф.
- Монтажнику не показывается лишняя аналитика.
- Монтажник не начисляет монеты вручную.

### Доступные экраны
- `000_Login`
- `001_Worker_Main`
- `002_Start_Work_Confirm`
- `003_Start_Work_Camera`
- `004_Task_Detail`
- `005_Take_Task_Confirm`
- `006_Task_In_Progress`
- `007_Step_Work`
- `008_Step_Confirm`
- `009_Step_Done`
- `010_Cannot_Do`
- `011_Finish_Work`
- `012_Worker_Messages`
- `013_Worker_Profile`
- `014_Task_History`
- системные экраны ошибок, загрузки, пустых состояний и начисления монет.

### Основные действия
- Авторизоваться.
- Начать смену.
- Подтвердить действие фото.
- Взять задачу.
- Выполнить этап.
- Завершить задачу.
- Завершить смену.
- Открыть сообщения и профиль.

## Роль: Прораб

### Обязанности
- Управлять сотрудниками и задачами.
- Создавать задачи.
- Назначать одного или нескольких исполнителей.
- Создавать этапы с названием, описанием и необязательным фото-примером.
- Контролировать, кто на работе, кто ушел и кто чем занимается.
- Коммуницировать с сотрудниками.

### Права
- Видеть список сотрудников.
- Видеть статус сотрудника, текущую задачу, прогресс, монеты и время.
- Создавать задачи.
- Назначать исполнителей.
- Указывать срок выполнения или оставлять его пустым.
- Создавать этапы задачи.
- Писать сотрудникам в чате.
- Просматривать черновики задач.

### Ограничения
- Прораб не начисляет монеты вручную.
- Прораб не принимает финансовое решение по выплатам.
- Первый экран прораба не должен быть объектным дашбордом.
- Прораб не должен видеть интерфейсы, предназначенные только для финансиста, если это не предусмотрено ролью.

### Доступные экраны
- `015_Foreman_Employees`
- `016_Foreman_Employee_Card`
- `017_Task_Create`
- `018_Task_Step_Builder`
- `019_Foreman_Messages`
- `020_Foreman_Drafts`
- системные экраны камеры, загрузки, пустых задач, доступности задач и завершения задач.

### Основные действия
- Открыть список сотрудников.
- Проверить статус сотрудника.
- Создать задачу.
- Добавить этапы.
- Назначить исполнителя или нескольких исполнителей.
- Написать сотруднику.
- Просмотреть черновики.

## Роль: Финансист

### Обязанности
- Анализировать монеты, рабочее время, задачи и историю активности.
- Сравнивать начисленные монеты с AI-оценкой стоимости выполненной работы.
- Рассматривать рекомендации ИИ.
- Принимать финальное решение по премиям и выплатам.
- Проверять историю выплат.
- Формировать финансовые отчеты.
- Следить за отклонениями и уведомлениями.

### Права
- Видеть общую сумму собранных монет.
- Видеть аналитику по дням, объектам и сотрудникам.
- Открывать карточку сотрудника.
- Смотреть диаграмму Ганта.
- Смотреть AI-рекомендации.
- Указывать размер премии и комментарий.
- Утверждать выплату в пределах бизнес-процесса.
- Смотреть историю выплат и отчеты.
- Настраивать финансовые фильтры, уведомления и режим просмотра.

### Ограничения
- Финансист не создает задачи как прораб.
- Финансист не меняет начисленные монеты вручную.
- Финансист не должен принимать решение без доступа к фактам и истории.
- AI-рекомендация не является автоматическим решением.
- Повторная выплата за тот же период и объект не допускается.

### Доступные экраны
- `029_Finance_Main`
- `030_Total_Coins`
- `031_Employee_Analytics`
- `032_AI_Recommendations`
- `033_Finance_Gantt`
- `034_Employee_Details`
- `035_Payments`
- `036_Payment_History`
- `037_Efficiency_Analytics`
- `038_Financial_Reports`
- `039_Finance_Notifications`
- `040_Finance_Settings`

### Основные действия
- Открыть главную финансиста.
- Проверить сумму монет и работающих сотрудников.
- Проанализировать сотрудника.
- Изучить AI-рекомендацию.
- Указать премию.
- Добавить комментарий.
- Подтвердить выплату.
- Выгрузить отчет.
- Настроить уведомления и фильтры.

## Роль: Руководитель

### Обязанности
- Оценивать эффективность компании, объектов и ролей.
- Контролировать прозрачность выполнения работ.
- Использовать данные для управленческих решений.
- Следить за качеством процессов и справедливостью мотивации.

### Права
- Получать агрегированную картину по объектам, людям, задачам и финансам.
- Проверять источники данных через историю задач, смен, фото и выплат.
- Использовать отчеты и метрики для управленческого анализа.

### Ограничения
- Руководитель не должен обходить ролевые правила без явной бизнес-логики.
- Управленческий вывод должен иметь связь с проверяемыми данными.
- AI-оценка остается рекомендацией, а не автоматическим решением.

### Доступные экраны
В текущем MVP отдельные экраны руководителя не выделены. Руководительская аналитика должна строиться на проверяемых данных UI Bible и финансовых отчетах, если такая роль будет формализована в следующих пакетах документации.

### Основные действия
- Смотреть сводные метрики.
- Проверять отклонения.
- Анализировать эффективность объектов.
- Принимать организационные решения на основе данных.

## Общие правила доступа
- API должен проверять роль пользователя на каждое защищенное действие.
- UI не должен показывать действия, недоступные роли.
- Серверная проверка обязательна даже если кнопка скрыта в интерфейсе.
- История действий должна сохранять пользователя, роль, время и связанную сущность.
- Роль не должна получать больше данных, чем требуется для ее задачи.
