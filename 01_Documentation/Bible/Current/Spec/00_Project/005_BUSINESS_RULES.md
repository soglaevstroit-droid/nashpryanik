# 005 — Business Rules

## Назначение
Документ фиксирует бизнес-правила НАШПРЯНИК.РФ. Эти правила нельзя менять через локальное решение экрана, API или реализации без обновления фундаментальной документации.

## Жизненный цикл задачи

### 1. Создание задачи
Задачу создает прораб. Задача может иметь одного исполнителя, нескольких исполнителей или быть свободной для доступных сотрудников объекта.

Задача содержит:
- название;
- место или объект, если оно доступно в контексте;
- описание;
- исполнителей, если они назначены;
- срок выполнения, если он нужен;
- этапы.

### 2. Этапы задачи
Каждый этап содержит название, описание и необязательное фото-пример. Этапы задают понятную последовательность выполнения и уменьшают риск неправильной работы.

Этап считается завершенным только после действия сотрудника и обязательной фиксации, если фото требуется логикой экрана.

### 3. Доступность задачи
Свободную задачу может взять любой монтажник, которому она доступна по объекту и статусу. Если задача назначена конкретному сотруднику, взять ее может только назначенный исполнитель.

Если задача уже в работе, другие сотрудники могут только просматривать ее, если это разрешено экраном.

### 4. Задача в работе
Монтажник может выполнять только одну задачу одновременно. Когда сотрудник взял задачу, остальные задачи приглушаются и доступны только для просмотра.

### 5. Завершение задачи
Задача завершается после выполнения необходимых этапов и подтверждений. Завершение должно попадать в историю с временем, сотрудником и связанными данными.

## Рабочая смена

### Статусы смены
Монтажник может быть в статусе:
- `На работе`;
- `Ушел`.

### Начало смены
Начало смены фиксирует, что сотрудник находится в рабочем состоянии. После начала смены система может начислять монеты за подтвержденные действия по правилам проекта.

### Завершение смены
Завершение смены переводит сотрудника в статус `Ушел`. После завершения смены новые монеты не должны начисляться до следующего начала работы.

### Главное правило начисления
Монеты начисляются только в статусе `На работе`.

## Фотофиксация
Все ключевые действия требуют фото. Фото является доказательством действия, состояния работы или причины невозможности выполнить задачу.

Фото используется для:
- подтверждения начала работы, если это требуется сценарием;
- подтверждения выполнения этапа;
- подтверждения завершения задачи;
- объяснения невозможности выполнить задачу;
- истории качества и аудита.

Фото отправляется через защищенный процесс загрузки. Состояния загрузки, ошибки и повторная отправка должны быть обработаны в UI и API.

## Статусы

### Статусы сотрудника
- `На работе` — сотрудник находится в активной смене.
- `Ушел` — сотрудник не находится в активной смене.

### Статусы задачи
- Свободная.
- Назначенная.
- В работе.
- Завершенная.
- Недоступная.
- Отмененная, если такой статус используется реализацией.

### Статусы этапа
- Не начат.
- В работе.
- Завершен.
- Невозможно выполнить, если сотрудник зафиксировал причину.

### Статусы выплаты
- Черновик.
- Готова к решению.
- Утверждена финансистом.
- Отклонена или отложена, если это предусмотрено процессом.

## Начисление монет
Монеты — внутренняя цифровая единица оценки труда. Они отражают подтвержденную работу и используются для аналитики, рекомендаций ИИ, премий и отчетов.

Правила:
- монеты не являются декоративной игровой механикой;
- монеты начисляются только за подтвержденные действия;
- монеты начисляются только в статусе `На работе`;
- прораб не начисляет монеты вручную;
- финансист не редактирует начисленные монеты вручную;
- исторические начисления должны оставаться проверяемыми.

## Рекомендации ИИ
ИИ анализирует выполненные задачи, рабочее время, историю активности, начисленные монеты, сложность работ и контекст объекта, если эти данные доступны.

ИИ может:
- сравнить начисленные монеты со своей оценкой стоимости работы;
- рекомендовать премию;
- подсветить отклонение;
- объяснить причину рекомендации;
- указать на событие, требующее внимания.

ИИ не может:
- начислять монеты;
- утверждать премии;
- создавать выплату без действия финансиста;
- менять бизнес-правила;
- скрывать данные;
- принимать финальное решение.

## Премии
Премия — дополнительная сумма, которую финансист может указать на основании данных, AI-рекомендации и собственного решения.

Правила:
- рекомендация ИИ не равна утвержденной премии;
- финансист может оставить премию равной нулю;
- комментарий финансиста фиксирует причину решения, если премия отличается от рекомендации или этого требует бизнес-процесс;
- итоговая сумма равна начисленным монетам плюс премия;
- повторная выплата за тот же период и объект не допускается.

## Финансовая история
История выплат хранит значения на момент утверждения. Если позже изменились профиль сотрудника, объект или правила, старая запись не пересчитывается автоматически.

История должна содержать:
- сотрудника;
- объект;
- период;
- начисленные монеты;
- рекомендацию ИИ;
- премию;
- итог;
- кто утвердил;
- дату выплаты.

## Уведомления
Уведомления помогают финансисту увидеть события, требующие внимания:
- новые рекомендации ИИ;
- большие отклонения;
- сотрудники без активности;
- подозрительные действия;
- готовые выплаты;
- события, требующие проверки.

Уведомление не выполняет финансовое действие само по себе.

## Ограничения системы
- Нельзя менять утвержденную бизнес-логику локально в одном экране.
- Нельзя давать ИИ право автоматически утверждать выплаты.
- Нельзя показывать монтажнику лишнюю аналитику и тарифы.
- Нельзя использовать другие цвета и стили вместо Design System.
- Нельзя создавать финансовые решения без истории.
- Нельзя считать неполные данные точными.
- Нельзя обходить серверную проверку ролей.

## Аудит и проверяемость
Каждое значимое действие должно иметь:
- автора;
- роль;
- время;
- связанную сущность;
- старое и новое состояние, если действие изменяет данные;
- источник данных, если действие связано с расчетом.
