# 006 — Success Metrics

## Назначение
Документ определяет KPI проекта НАШПРЯНИК.РФ. Метрики нужны, чтобы оценивать не только факт разработки, но и пользу продукта для строительной компании, сотрудников, интерфейса и качества системы.

## Продуктовые KPI

### Активное использование
Система успешна, если ключевые роли используют ее в ежедневной работе:
- монтажники начинают и завершают смены;
- монтажники берут задачи через приложение;
- прорабы создают задачи и этапы;
- финансисты анализируют монеты и выплаты;
- история действий формируется автоматически.

### Доля задач с полной историей
Целевой показатель — максимальная доля задач, у которых есть исполнитель, этапы, статусы, время, фото и история завершения.

### Снижение спорных ситуаций
Успех продукта проявляется в снижении числа споров по вопросам:
- кто выполнял работу;
- когда работа была выполнена;
- почему задача не завершена;
- за что начислены монеты;
- почему рекомендована премия.

### Прозрачность выплат
Финансовое решение считается качественным, если по нему можно восстановить начисленные монеты, AI-рекомендацию, премию, комментарий, утверждающего и дату.

## KPI сотрудников

### Для монтажника
- количество завершенных задач;
- количество завершенных этапов;
- подтвержденное рабочее время;
- заработанные монеты;
- доля задач с корректной фотофиксацией;
- количество ситуаций `Невозможно выполнить` с объяснением;
- отсутствие параллельных задач в работе.

### Для прораба
- количество созданных задач;
- доля задач с понятными этапами;
- доля задач с назначенными исполнителями, если назначение требуется;
- скорость реакции на сотрудников без активности;
- доля задач без просрочек;
- качество коммуникации через сообщения;
- отсутствие ручного начисления монет.

### Для финансиста
- доля выплат с сохраненной историей решения;
- доля премий с комментарием, если комментарий требуется;
- скорость обработки готовых выплат;
- количество просмотренных AI-рекомендаций;
- количество проверенных отклонений;
- корректность итоговой суммы выплат.

## KPI интерфейса

### Простота сценариев
Ключевые сценарии должны выполняться без лишних шагов:
- начать работу;
- взять задачу;
- выполнить этап;
- подтвердить фото;
- завершить работу;
- создать задачу;
- принять решение по выплате.

### Мобильная пригодность
Все основные экраны проектируются сначала под iPhone. Успешный экран:
- не имеет горизонтального скролла страницы;
- сохраняет читаемость;
- не ломает композицию при пустых и ошибочных состояниях;
- показывает главное действие без поиска.

### Сохранение Design System
Каждый экран должен:
- повторять утвержденный референс минимум на 95%;
- использовать фирменный оранжевый только для ключевых акцентов;
- использовать зеленый только для успеха и статуса `На работе`;
- использовать красный только для ошибок и опасных действий;
- не вводить новые стили карточек без необходимости.

### Понятность
Экран считается понятным, если пользователь может объяснить:
- что сейчас происходит;
- какое главное действие доступно;
- почему действие недоступно;
- что изменится после нажатия.

## KPI качества данных

### Полнота данных
Для задач, смен, этапов, монет и выплат должны быть заполнены обязательные поля. Пустые значения допустимы только там, где это разрешено бизнес-логикой.

### Связность данных
Финансовые и рабочие показатели должны связываться с сущностями:
- сотрудник;
- объект;
- задача;
- этап;
- смена;
- начисление;
- рекомендация;
- выплата.

### Историчность
История должна позволять восстановить прошлое состояние. Утвержденные выплаты не пересчитываются задним числом без явного процесса.

### Обработка ошибок
Качество системы включает корректную обработку:
- Loading;
- Empty;
- Ready;
- Error;
- Offline;
- Saving;
- Uploading photo;
- Disabled action;
- Conflict.

## KPI AI-логики

### Объяснимость
Каждая AI-рекомендация должна показывать основание: какие данные повлияли на вывод и почему рекомендация появилась.

### Контроль человеком
Ни одна AI-рекомендация не должна автоматически утверждать выплату, менять монеты или скрывать событие. Успех AI-функции измеряется полезностью подсказки, а не автономностью решения.

### Качество сигналов
AI-сигналы должны помогать находить:
- большие отклонения;
- возможные премии;
- подозрительные действия;
- сотрудников без активности;
- несоответствия между фактом и ожиданиями.

## KPI инженерного качества

### Соответствие документации
Реализация должна соответствовать:
- `MASTER_PROMPT.md`;
- `PROJECT.md`;
- `00_Project`;
- UI Bible;
- Design System;
- API-документации;
- бизнес-правилам.

### Проверяемость
Ключевые сценарии должны иметь критерии приемки. Если действие влияет на деньги, статусы, фото или историю, оно требует повышенного внимания в тестировании.

### Минимум скрытой логики
Расчеты монет, премий, отклонений и итогов должны быть описаны и проверяемы. Нельзя оставлять важную бизнес-логику только в коде без документации.

## KPI выпуска
Функциональность готова к выпуску, если:
- реализован основной сценарий;
- обработаны состояния загрузки, пустоты, ошибки и недоступности;
- соблюден Design System;
- нет нарушения ролевых прав;
- финансовые действия сохраняют историю;
- AI не принимает финальное решение;
- экран работает на iPhone;
- критерии приемки выполнены.
