# DEVELOPMENT RULES

## Проект: СТРОИТ.РФ

Версия: 1.0

Статус: APPROVED

---

# Назначение

Настоящий документ определяет единые правила разработки проекта СТРОИТ.РФ.

Все разработчики, архитекторы и AI-инструменты (включая Codex) обязаны придерживаться данных правил.

Главная цель — обеспечить единый стиль разработки, высокое качество кода и предсказуемость развития проекта.

---

# Основные принципы

1. Архитектура важнее скорости разработки.
2. Документация является частью продукта.
3. Код обязан соответствовать документации.
4. Простота предпочтительнее сложности.
5. Любое изменение должно быть понятным другому разработчику.
6. Решения принимаются с учетом масштабирования проекта.
7. Код должен быть читаемым, а не "умным".
8. Любое изменение должно иметь понятную причину.

# Правила коммитов

Формат:

type(scope): description

Примеры:

feat(worker): add task details screen

feat(api): implement authentication

fix(photo): resolve upload timeout

docs(backend): update architecture

refactor(auth): simplify jwt validation

test(api): add integration tests

# Pull Request

Каждый Pull Request должен содержать:

- цель изменения;
- список измененных файлов;
- влияние на архитектуру;
- связанные ADR;
- связанные документы;
- результаты тестирования.

---

# Code Review

Проверяется:

- читаемость;
- архитектура;
- безопасность;
- производительность;
- соответствие документации;
- отсутствие дублирования;
- качество именования.

---

# Правила именования

Файлы:

snake_case

PascalCase

camelCase

UPPER_CASE

Endpoint:

kebab-case

---

# Документация

Перед изменением архитектуры необходимо обновить:

- Bible;
- ADR;
- API;
- Database;
- Backend;
- Frontend;

если изменения затрагивают соответствующие разделы.

---

# Запрещается

- писать код без понятной причины;
- создавать дублирующую функциональность;
- нарушать утвержденную архитектуру;
- использовать "магические числа";
- хранить секреты в репозитории;
- изменять Production вручную.

---

# Testing

Каждая новая функция должна иметь:

- Unit Test (если применимо);
- Integration Test (для Backend);
- Проверку пользовательского сценария.

---

# Definition of Done

Функция считается завершенной только если:

✓ код написан;

✓ код соответствует архитектуре;

✓ документация обновлена;

✓ тесты проходят;

✓ нет ошибок линтера;

✓ Pull Request проверен;

✓ код смержен в develop.

---

# Работа с AI

AI помогает:

- писать код;
- анализировать архитектуру;
- искать ошибки;
- создавать документацию;
- предлагать улучшения.

AI не принимает самостоятельных инженерных решений.

Все архитектурные решения утверждаются человеком.

---

# Принцип разработки

Перед написанием любого нового кода разработчик обязан ответить на три вопроса:

1. Зачем нужна эта функция?
2. Соответствует ли она архитектуре?
3. Будет ли она полезна через пять лет?

Если хотя бы на один вопрос ответ отрицательный — решение должно быть пересмотрено.

---

# Заключение

Настоящие правила являются обязательными для всех участников разработки.

Цель данных правил — не ограничить разработчиков, а обеспечить создание надежной, понятной и масштабируемой платформы.

Мы строим систему, которая должна развиваться многие годы, поэтому качество решений всегда важнее скорости их реализации.
