# MASTER DEVELOPMENT ROADMAP — НАШПРЯНИК.РФ MVP

Версия: 1.0

Статус: основной roadmap перехода от документации к разработке

Дата: 2026-07-06

---

## 1. Назначение документа

Документ определяет последовательность инженерной реализации MVP после завершения Product Bible, UI Bible, Architecture Bible, Design System, Database Bible, API Bible, Backend Bible, Frontend Bible и Documentation Audit.

Roadmap не вводит новую бизнес-логику, роли, статусы или сущности. Все решения опираются на существующую документацию проекта и `AUDIT_REPORT.md`.

Главная задача roadmap — превратить утвержденную документацию в управляемую последовательность разработки: от подготовки репозитория до MVP release.

## 2. Статус проекта

На момент старта разработки подготовлены и согласованы:

- Product Bible: цели, принципы, роли, бизнес-правила, метрики успеха и глоссарий.
- Design System: цвета, типографика, компоненты, состояния и визуальные ограничения.
- UI Bible: утвержденные mobile first сценарии для монтажника, прораба и финансиста.
- Architecture Bible: модули, навигация, state machine, data flow, event flow и масштабирование.
- Database Bible: PostgreSQL-модель, связи, enum, аудит, индексы и правила масштабирования.
- API Bible: REST API `/api/v1`, JWT, RBAC, ошибки, multipart upload, endpoint и JSON-примеры.
- Backend Bible: Service + Repository архитектура, middleware, storage, AI, уведомления, безопасность и deployment.
- Frontend Bible: Flutter mobile architecture, routing, state management, API client, offline, camera, upload и release build.
- Documentation Audit: проверка согласованности между ключевыми разделами, исправление выявленных технических несостыковок и нулевой остаток замечаний.

Документация считается достаточной для перехода к реализации MVP при условии, что код не расширяет продуктовую модель без отдельного изменения источников истины.

## 3. Цель MVP

MVP должен дать строительной компании рабочее приложение, в котором:

- монтажник начинает и завершает смену, берет доступную задачу, выполняет этапы, подтверждает действия фото и видит собственные монеты;
- прораб видит сотрудников, их рабочий статус и текущие задачи, создает задачи и этапы, назначает исполнителей и работает с черновиками;
- финансист анализирует монеты, сотрудников, объекты, отклонения, AI-рекомендации, выплаты, историю и отчеты;
- backend проверяет роли, статусы, права, целостность данных и финансовые ограничения независимо от UI;
- PostgreSQL хранит структурированные данные, связи, историю и аудит;
- файловое хранилище хранит фотографии, а PostgreSQL хранит метаданные и ссылки;
- AI Engine формирует объяснимые рекомендации, но не утверждает выплаты, не начисляет монеты и не меняет статусы;
- приложение сохраняет утвержденный Design System и повторяет UI-референсы минимум на 95%.

## 4. Роли MVP

Утвержденные роли разработки:

- `worker` — монтажник;
- `foreman` — прораб;
- `finance` — финансист;
- `manager` — руководительская роль без отдельного набора MVP-экранов.

Отдельный admin UI, admin routes и admin permissions не входят в MVP, потому что роль администратора не введена в Product Bible, Database Bible, API Bible, Backend Bible и Documentation Audit. Если такая роль потребуется позже, она должна быть сначала добавлена в фундаментальную документацию.

## 5. Принципы перехода от документации к коду

1. Документация является источником истины, код является реализацией.
2. Любая реализация начинается с проверки соответствующих Bible-документов.
3. Если код требует новой бизнес-логики, разработка останавливается до обновления документации.
4. Backend всегда проверяет роль, статус и права, даже если frontend скрывает действие.
5. Frontend не рассчитывает финальные бизнес-решения, а отображает состояния и вызывает API.
6. Database гарантирует целостность, историю, связи и невозможность очевидных дублей.
7. AI работает только с подтвержденными данными и возвращает рекомендацию с объяснением.
8. Финансовые действия сохраняются в истории и audit log.

## 6. Последовательность разработки MVP

### 6.1 Repository Cleanup

Цель: привести репозиторий к структуре, в которой backend, frontend, документация, конфигурация и deployment не конфликтуют друг с другом.

Результат:

- определена структура monorepo или согласованная структура каталогов;
- удалены устаревшие артефакты, не относящиеся к MVP;
- зафиксированы команды запуска, проверки и сборки;
- документация остается неизмененной источником истины.

### 6.2 Backend Foundation

Цель: создать серверный каркас, соответствующий Backend Bible и API Bible.

Результат:

- Express backend с модульной структурой;
- контроллеры, сервисы и репозитории разделены;
- единый error handler;
- базовые middleware;
- конфигурация окружений;
- logging и request id.

### 6.3 Database Implementation

Цель: реализовать PostgreSQL-модель из Database Bible.

Результат:

- миграции для компаний, пользователей, объектов, задач, этапов, смен, фото, монет, уведомлений, AI-рекомендаций и audit log;
- enum соответствуют `013_ENUMS.md`;
- связи и индексы соответствуют `003_RELATIONSHIPS.md` и `015_INDEXING.md`;
- модель поддерживает множество компаний, объектов и пользователей.

### 6.4 Auth & RBAC

Цель: реализовать JWT-аутентификацию и role-based access control.

Результат:

- login, refresh, logout;
- защищенные routes;
- middleware проверки JWT;
- middleware проверки роли;
- запрет доступа к чужой компании;
- единый формат ошибок `401`, `403`, `409` и `422`.

### 6.5 Task Engine

Цель: реализовать жизненный цикл задач, этапов и смен.

Результат:

- создание задач прорабом;
- этапы задач;
- назначение исполнителей;
- взятие свободной задачи монтажником;
- ограничение одной активной задачи;
- статусы смены `На работе` и `Ушел`;
- история действий.

### 6.6 Photo Engine

Цель: реализовать фотофиксацию ключевых действий.

Результат:

- `multipart/form-data` upload;
- storage adapter;
- таблица metadata `photos`;
- статусы загрузки;
- связь фото с задачей, этапом, сменой или причиной невозможности выполнения;
- retry и ошибки загрузки.

### 6.7 Coin Engine

Цель: реализовать начисление и отображение монет в границах утвержденных правил.

Результат:

- начисление только за подтвержденную работу;
- начисление только в статусе `На работе`;
- запрет ручного начисления прорабом и финансистом;
- сохранение транзакций монет;
- аналитика монет для финансовых экранов.

### 6.8 Worker App

Цель: реализовать Flutter-сценарии монтажника.

Результат:

- login;
- главная монтажника;
- старт и завершение смены;
- камера прихода и рабочих действий;
- задача, этапы, подтверждения, состояние `Не вывожу`;
- сообщения, профиль, история и системные состояния.

### 6.9 Foreman App

Цель: реализовать Flutter-сценарии прораба.

Результат:

- список сотрудников;
- карточка сотрудника;
- создание задачи;
- step builder;
- назначение исполнителей;
- сообщения;
- черновики;
- проверка ограничений роли.

### 6.10 Finance App

Цель: реализовать Flutter-сценарии финансиста.

Результат:

- главная финансиста;
- total coins;
- employee analytics;
- AI recommendations;
- Gantt;
- employee details;
- payments;
- payment history;
- efficiency analytics;
- financial reports;
- notifications;
- settings.

### 6.11 AI Recommendations

Цель: реализовать интеграцию с AI Engine как рекомендательным сервисом.

Результат:

- backend собирает подтвержденные данные;
- AI возвращает оценку, разницу, тип рекомендации, confidence и объяснение;
- результат сохраняется в `ai_recommendations`;
- при необходимости создается уведомление;
- AI не утверждает выплаты и не меняет монеты.

### 6.12 Notifications

Цель: реализовать уведомления для событий, требующих внимания.

Результат:

- in-app notifications;
- типы уведомлений из Database Bible;
- статусы `new`, `viewed`, `resolved`, `hidden`, `error`;
- связь с AI-рекомендациями, отклонениями, сотрудниками без активности и готовыми выплатами;
- уведомление не выполняет финансовое действие само по себе.

### 6.13 Testing

Цель: подтвердить, что MVP соответствует документации.

Результат:

- backend unit и integration tests;
- API contract tests;
- database migration checks;
- Flutter widget и integration tests;
- ручная проверка UI-сценариев;
- проверка 95% соответствия утвержденным референсам;
- проверка безопасности ролей и финансовых действий.

### 6.14 Deployment

Цель: подготовить воспроизводимый запуск MVP.

Результат:

- окружения development и production;
- переменные окружения;
- PostgreSQL migrations;
- storage configuration;
- сборка Flutter release;
- healthcheck;
- logging;
- backup strategy.

### 6.15 MVP Release

Цель: выпустить первую рабочую версию без расширения утвержденного объема.

Результат:

- все ключевые сценарии проходят проверку;
- критические ошибки закрыты;
- документация и код синхронизированы;
- рабочие данные сохраняют историю и аудит;
- release notes отражают только реализованные возможности MVP.

## 7. Спринты

### Sprint 1 — Repository Cleanup и инженерный каркас

Цель: подготовить репозиторий к разработке.

Задачи:

- зафиксировать структуру backend, frontend, docs и deployment;
- убрать или изолировать устаревшие артефакты;
- настроить команды запуска и проверки;
- подготовить базовую конфигурацию окружений.

Результат:

- репозиторий готов к разработке без конфликтов старого и нового MVP.

Критерии готовности:

- ветка разработки собирается;
- команды запуска описаны;
- документация не изменена;
- нет неучтенных файлов и временных артефактов.

### Sprint 2 — Backend Foundation и Database Base

Цель: создать основу backend и PostgreSQL.

Задачи:

- реализовать Express application shell;
- создать слои Controller, Service, Repository;
- подключить PostgreSQL;
- создать начальные миграции;
- реализовать healthcheck и единый error format.

Результат:

- backend запускается и подключается к базе.

Критерии готовности:

- healthcheck работает;
- миграции применяются на чистой базе;
- ошибки возвращаются в формате API Bible;
- структура соответствует Backend Bible.

### Sprint 3 — Auth, RBAC и multi-company isolation

Цель: защитить API и данные.

Задачи:

- реализовать JWT login, refresh и logout;
- реализовать middleware роли;
- реализовать company scope;
- покрыть доступы для `worker`, `foreman`, `finance`, `manager`;
- проверить запрет чужих данных.

Результат:

- защищенные endpoint не доступны без корректной роли и компании.

Критерии готовности:

- `401` и `403` возвращаются корректно;
- backend проверяет роль независимо от UI;
- пользователь не получает данные другой компании;
- audit фиксирует значимые auth-события.

### Sprint 4 — Task Engine и Work Sessions

Цель: реализовать рабочий цикл монтажника и управление задачами прораба.

Задачи:

- реализовать смены;
- реализовать задачи и этапы;
- реализовать назначение исполнителей;
- реализовать взятие задачи;
- реализовать ограничение одной активной задачи;
- реализовать историю статусов.

Результат:

- монтажник и прораб могут выполнять базовый рабочий процесс через API.

Критерии готовности:

- монеты не начисляются вне статуса `На работе`;
- монтажник не берет две задачи одновременно;
- назначенная задача доступна только разрешенному исполнителю;
- конфликтные действия возвращают `409`.

### Sprint 5 — Photo Engine и подтверждения

Цель: сделать фотофиксацию надежной частью workflow.

## 8. Зависимости

### Database

Database зависит от Product Bible, Architecture Bible и UI Bible. Backend и API зависят от миграций, enum, связей, индексов и audit log.

Критическая зависимость: auth, task engine, photo engine, coin engine, finance и AI не могут считаться готовыми без корректной PostgreSQL-модели.

### API

API зависит от Database Bible и Backend Bible. Frontend зависит от стабильных endpoint, JSON-форматов, error codes, RBAC и multipart upload.

Критическая зависимость: Flutter-экраны не должны обходить API или реализовывать собственную бизнес-логику.

### Backend

Backend зависит от API Bible, Database Bible и Architecture Bible. AI, notifications, finance и frontend зависят от backend как единственной точки проверки ролей и статусов.

Критическая зависимость: сервисный слой должен быть реализован до сложных UI-интеграций, чтобы frontend не строился на временных контрактах.

### Frontend

Frontend зависит от UI Bible, Design System, API Bible и Backend Bible. Frontend может начинаться с mocked API только если контракты совпадают с `04_API`.

Критическая зависимость: mobile screens должны проверяться на реальных состояниях Loading, Empty, Ready, Error, Offline, Saving, Uploading photo, Disabled action и Conflict.

### AI

AI зависит от подтвержденных данных backend, PostgreSQL-истории, монет, выполненных задач, смен и контекста объекта.

Критическая зависимость: AI-рекомендации нельзя подключать к выплатам до готовности audit log и finance API.

### DevOps

DevOps зависит от backend configuration, database migrations, storage configuration, logging, healthcheck и Flutter release build.

Критическая зависимость: production deployment нельзя считать готовым без backup strategy и проверки миграций на чистом окружении.

## 9. Definition of Done

### Для документации

- документ не противоречит Product Bible, UI Bible, Design System, Architecture Bible, Database Bible, API Bible, Backend Bible и Frontend Bible;
- не вводит новые роли, статусы, сущности или бизнес-правила;
- не содержит запрещенных шаблонных маркеров;
- содержит проверяемые критерии готовности.

### Для backend

- endpoint реализованы по API Bible;
- сервисы не содержат UI-логики;
- репозитории работают через PostgreSQL-модель;
- JWT и RBAC применяются на защищенных routes;
- ошибки возвращаются в едином формате;
- финансовые действия пишутся в audit log.

### Для frontend

- Flutter-экраны соответствуют UI Bible и Design System;
- role routing не показывает чужие сценарии;
- API client использует `/api/v1`;
- offline queue не создает дубли;
- camera и photo upload обрабатывают ошибки;
- состояния Loading, Empty, Ready, Error, Offline, Saving, Uploading photo, Disabled action и Conflict реализованы.

### Для UI

- экран mobile first;
- референс повторен минимум на 95%;
- главное действие понятно;
- недоступное действие объясняется состоянием;
- цвета и компоненты соответствуют Design System;
- текст не ломает layout на мобильном экране.

### Для API

- endpoint имеет метод, путь, роли, параметры, request и response examples;
- endpoint связан с таблицами БД и UI-сценарием;
- ошибки используют единый формат;
- `multipart/form-data` применяется для фотографий;
- версии начинаются с `/api/v1`.

### Для БД

- миграции применяются на чистой базе;
- связи и индексы соответствуют Database Bible;
- `company_id` используется для multi-company данных;
- исторические данные не удаляются физически без описанного процесса;
- audit log фиксирует значимые действия.

### Для релиза

- основные сценарии монтажника, прораба и финансиста проходят вручную;
- automated checks проходят;
- production configuration подготовлена;
- миграции воспроизводимы;
- backup и healthcheck проверены;
- release notes соответствуют фактически реализованному объему.

## 10. Риски

### Рассинхронизация документации и кода

Риск: код начинает жить отдельно от Bible-документов.

Снижение риска:

- code review проверяет ссылку на источник истины;
- изменения бизнес-логики сначала проходят через документацию;
- audit повторяется перед MVP release.

### Слабая фотофиксация

Риск: ключевое действие не подтверждено фото или фото невозможно связать с сущностью.

Снижение риска:

- upload metadata сохраняется транзакционно с доменным действием там, где это требуется сценарием;
- UI показывает состояния загрузки и ошибки;
- backend валидирует тип, размер, автора, компанию и связанную сущность.

### Ошибки начисления монет

Риск: монеты начисляются вне статуса `На работе`, повторно или вручную не той ролью.

Снижение риска:

- начисление проходит через CoinService;
- проверяется активная смена;
- транзакции монет сохраняют источник;
- роли `foreman` и `finance` не получают ручного начисления.

### Сложность UI

Риск: интерфейс становится тяжелым для пользователя без технического образования.

Снижение риска:

- реализация строго повторяет UI Bible;
- каждый экран проверяется на главное действие;
- Design System не расширяется локальными стилями;
- ручная проверка проводится на мобильных viewport.

### Нестабильная работа на телефоне

Риск: камера, upload, offline queue или длинные списки работают нестабильно.

Снижение риска:

- проверка на реальном мобильном сценарии;
- lazy loading для списков;
- retry для загрузки фото;
- контроль потребления памяти и сетевых ошибок.

### Проблемы синхронизации

Риск: offline-действия создают дубли задач, этапов, фото или выплат.

Снижение риска:

- idempotency для критических действий;
- конфликтные состояния возвращают `409`;
- платежные действия не повторяются автоматически;
- локальная очередь хранит только допустимые операции.

### Ошибки AI-рекомендаций

Риск: AI дает неверную или непонятную рекомендацию.

Снижение риска:

- AI получает только подтвержденные данные;
- рекомендация сохраняет explanation и confidence;
- финансист принимает финальное решение;
- AI-сбой не ломает сценарии монтажника и прораба.

## 11. Контроль качества

### Documentation Audit

Перед началом крупных этапов и перед MVP release проверяется согласованность документации и кода:

- роли;
- статусы;
- API и Database;
- Backend и API;
- Frontend и API;
- UI и Backend;
- Design System usage;
- отсутствие запрещенных шаблонных маркеров.

### Code Review

Каждая change request проверяется по вопросам:

- какой Bible-документ является источником решения;
- не добавлена ли новая бизнес-логика;
- не расширены ли роли или статусы;
- сохранены ли RBAC и company scope;
- есть ли тесты для финансовых, фото и статусных действий.

### Tests

Обязательные группы проверок:

- backend unit tests для сервисов;
- repository tests для PostgreSQL-запросов;
- API integration tests;
- migration tests;
- Flutter widget tests;
- Flutter integration tests для ключевых сценариев;
- regression tests для ролей и финансовых действий.

### Manual UI Review

Ручная проверка проводится по UI Bible:

- экран соответствует референсу минимум на 95%;
- mobile layout не ломается;
- states реализованы;
- главное действие видно;
- запрещенные действия недоступны.

### API Review

Проверяется:

- путь начинается с `/api/v1`;
- request и response соответствуют API Bible;
- ошибка возвращается в едином формате;
- роль проверяется backend;
- endpoint не раскрывает данные другой компании.

### Database Review

Проверяется:

- миграции применяются на чистой базе;
- foreign keys и indexes соответствуют Database Bible;
- enum не расходятся со статусами UI и Architecture;
- audit log содержит значимые действия;
- финансовая история восстановима.

Следующие задачи должны быть сформированы как отдельные TASK-файлы перед началом реализации:

## 13. Порядок старта разработки

Первым рабочим шагом после этого roadmap должен быть `TASK_012_REPOSITORY_CLEANUP`. До него не следует начинать feature-разработку, потому что backend, frontend, database и deployment должны опираться на чистую структуру репозитория.

После cleanup разработка идет в порядке зависимостей:

1. backend foundation;
2. database implementation;
3. auth and RBAC;
4. task engine;
5. photo engine;
6. coin engine;
7. worker app;
8. foreman app;
9. finance app;
10. AI recommendations;
11. testing;
12. deployment;
13. MVP release.

Этот порядок защищает проект от ситуации, когда UI, API и база начинают реализовывать разные версии одного сценария.

## 14. Итог

Документация проекта готова к переходу в разработку MVP. Следующий инженерный фокус — не расширять продукт, а последовательно реализовать уже утвержденные сценарии, контракты, роли, статусы, таблицы, endpoint, UI-состояния и критерии качества.

MVP считается успешным, если пользовательские сценарии монтажника, прораба и финансиста работают на реальных данных, финансовые решения проверяемы, фотофиксация надежна, AI остается рекомендательным инструментом, а код соответствует утвержденной документации.
