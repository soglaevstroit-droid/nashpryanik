# 003 — Relationships

## Назначение
Документ описывает связи между сущностями, их тип, обязательность, правила удаления и каскадные обновления.

## Company → Users
- Тип: 1:N.
- Обязательность: пользователь обязан принадлежать компании.
- FK: `users.company_id → companies.id`.
- Удаление: компания не удаляется физически, используется архивирование.
- Обновление: `company_id` не меняется после создания пользователя.

## Company → Objects
- Тип: 1:N.
- Обязательность: объект обязан принадлежать компании.
- FK: `objects.company_id → companies.id`.
- Удаление: объект архивируется, чтобы не разрушать историю задач и выплат.
- Обновление: каскадное обновление ключей не используется.

## Company → Domain Tables
- Тип: 1:N для задач, этапов, смен, фото, монет, уведомлений, AI-рекомендаций.
- Обязательность: `company_id` обязателен.
- Удаление: физическое удаление компании запрещено для активных данных.
- Назначение: предотвращает смешивание данных разных компаний.

## Object → Tasks
- Тип: 1:N.
- Обязательность: задача должна быть связана с объектом, если она относится к объектному списку работ.
- FK: `tasks.object_id → objects.id`.
- Удаление: объект архивируется, задачи сохраняются.
- Обновление: перенос задачи между объектами не должен выполняться без явного бизнес-процесса.

## User → Created Tasks
- Тип: 1:N.
- Обязательность: у задачи должен быть создатель.
- FK: `tasks.created_by → users.id`.
- Правило: создателем задачи является прораб.
- Удаление: пользователь не удаляется физически, используется деактивация.

## Task → TaskSteps
- Тип: 1:N.
- Обязательность: этап обязан принадлежать задаче.
- FK: `task_steps.task_id → tasks.id`.
- Удаление: задача не удаляется физически, этапы сохраняются.
- Каскад: физический cascade delete не используется для исторических данных.

## Task ↔ Users через TaskAssignees
- Тип: N:M.
- Обязательность: задача может быть свободной без исполнителей или иметь одного/нескольких назначенных исполнителей.
- FK: `task_assignees.task_id → tasks.id`, `task_assignees.user_id → users.id`.
- Удаление: назначение архивируется или удаляется только до появления исторических событий.
- Ограничение: пользователь должен принадлежать той же компании.

## User → WorkSessions
- Тип: 1:N.
- Обязательность: смена обязана иметь сотрудника.
- FK: `work_sessions.user_id → users.id`.
- Удаление: пользователь деактивируется, смены сохраняются.
- Ограничение: у сотрудника не должно быть более одной активной смены.

## Object → WorkSessions
- Тип: 1:N.
- Обязательность: объект обязателен, если смена привязана к объекту.
- FK: `work_sessions.object_id → objects.id`.
- Удаление: объект архивируется, смены сохраняются.

## Task/Step/Session/User → Photos
- Тип: 1:N от каждой сущности к фото.
- Обязательность: фото всегда имеет автора и компанию; связь с задачей, этапом или сменой зависит от типа фото.
- FK: `photos.uploaded_by → users.id`, `photos.task_id → tasks.id`, `photos.step_id → task_steps.id`, `photos.work_session_id → work_sessions.id`.
- Удаление: фото-метаданные сохраняются; файл может архивироваться по правилам storage.

## Step → Example Photo
- Тип: 1:1 optional.
- Обязательность: фото-пример необязательно.
- FK: `task_steps.example_photo_id → photos.id`.
- Удаление: фото-пример не удаляется физически, если используется этапом.

## User/Object/Task/Step/Session → CoinTransactions
- Тип: 1:N.
- Обязательность: транзакция монет обязана иметь сотрудника, компанию и источник; объект/задача/этап/смена обязательны, если начисление связано с работой.
- Удаление: транзакции не удаляются физически.
- Ограничение: `amount >= 0`, начисление возможно только из подтвержденного события backend.

## User/Role → Notifications
- Тип: 1:N.
- Обязательность: уведомление должно иметь получателя-пользователя или роль-получатель.
- Удаление: уведомления сохраняются для истории обработки.
- Связанные сущности хранятся как `related_entity_type` и `related_entity_id` с проверкой backend.

## User/Object → AIRecommendations
- Тип: 1:N.
- Обязательность: рекомендация должна быть связана с сотрудником; объект может быть указан, если рекомендация объектная.
- Удаление: рекомендации сохраняются как финансовая история.
- Ограничение: AI-рекомендация не утверждает выплату.

## AIRecommendation → Payment
- Тип: 1:N или 1:1 в рамках решения выплаты.
- Обязательность: выплата может ссылаться на рекомендацию, если решение принято по ней.
- Удаление: рекомендация сохраняется.
- Правило: финальное решение сохраняет финансист.

## AuditLog → All Domain Entities
- Тип: N:1 через `entity_type` и `entity_id`.
- Обязательность: каждое значимое действие должно иметь запись аудита.
- Удаление: audit log хранится согласно политике хранения и не удаляется вместе с сущностями.
