# 001 — Database Overview

## Назначение
Документ описывает архитектуру хранения данных НАШПРЯНИК.РФ. Модель проектируется под PostgreSQL как основную БД, файловое хранилище фотографий и масштабирование на множество компаний, объектов и пользователей.

## Основная БД
PostgreSQL хранит структурированные данные:
- компании;
- пользователей и роли;
- строительные объекты;
- задачи;
- исполнителей задач;
- этапы;
- рабочие смены;
- метаданные фото;
- транзакции монет;
- уведомления;
- AI-рекомендации;
- журнал аудита.

PostgreSQL является источником истины для связей, статусов, истории, финансовых расчетов и аналитики. Файлы фото не хранятся как бинарные данные в основных бизнес-таблицах.

## Файловое хранилище фотографий
Фотографии хранятся во внешнем файловом хранилище. Таблица `photos` хранит:
- ссылку на файл;
- тип фото;
- автора;
- компанию;
- объект;
- связанную сущность;
- статус загрузки;
- время создания.

Такое разделение позволяет масштабировать миллионы фото без раздувания PostgreSQL и сохранять проверяемые связи в базе.

## Multi-company модель
Для поддержки нескольких компаний вводится сущность `companies`. Все доменные таблицы, где данные принадлежат компании, содержат `company_id`.

Правило: данные разных компаний не должны смешиваться в запросах, отчетах, аналитике и AI-рекомендациях.

## Целостность данных
Целостность обеспечивается:
- внешними ключами;
- `NOT NULL` для обязательных полей;
- `CHECK` для неотрицательных сумм и корректных диапазонов;
- уникальными индексами для предотвращения дублей;
- partial unique index для одной активной смены и одной активной задачи у сотрудника;
- аудитом значимых действий.

## Нормализация
Модель нормализована вокруг доменных сущностей:
- пользователь отдельно от объекта;
- задача отдельно от этапов;
- фото отдельно от файлового storage;
- монеты отдельно от задач и этапов;
- AI-рекомендация отдельно от выплаты;
- уведомление отдельно от события.

Нормализация нужна, чтобы аналитика, финансы и история не расходились между экранами.

## Историчность
Исторические финансовые и рабочие данные должны оставаться восстановимыми. Утвержденные выплаты и начисленные монеты не пересчитываются задним числом автоматически.

Для удаления рабочих данных используется мягкое удаление через `deleted_at` или архивирование, если сущность уже участвует в истории.

## Рекомендуемые типы PostgreSQL
- `uuid` для первичных ключей.
- `timestamptz` для дат и времени.
- `numeric(14,2)` для финансово значимых сумм, если нужно хранить дробные значения.
- `integer` или `bigint` для монет, если монеты целочисленные.
- `jsonb` для объяснений AI, снимков контекста и безопасных расширяемых метаданных.
- enum-типы PostgreSQL для утвержденных статусов.

## Транзакционные границы
Операции, влияющие на бизнес-историю, выполняются транзакционно:
- взятие задачи;
- завершение этапа с фото;
- начисление монет;
- закрытие смены;
- создание AI-рекомендации;
- утверждение выплаты;
- изменение статуса уведомления.

## Архитектурное ограничение
База данных поддерживает бизнес-логику, но не заменяет backend. Backend отвечает за сценарные проверки, а PostgreSQL гарантирует целостность, уникальность и невозможность очевидно некорректных данных.
