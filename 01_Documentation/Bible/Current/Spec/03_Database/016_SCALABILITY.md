# 016 — Scalability

## Назначение
Документ описывает масштабирование базы данных на тысячи пользователей, множество объектов, несколько компаний, миллионы фотографий, архивирование и резервное копирование.

## Тысячи пользователей
Модель поддерживает рост пользователей через:
- `company_id` на всех доменных таблицах;
- индексы по роли, статусу и компании;
- partial unique index для одной активной смены;
- partial unique index для одной задачи в работе;
- порционную загрузку списков.

Главные экраны не должны читать всю историю пользователя. Детальные данные загружаются при открытии карточки или отчета.

## Большое количество объектов
Объект является измерением задач, смен, фото, монет, AI-рекомендаций и отчетов.

Для масштабирования:
- объекты архивируются, а не удаляются;
- исторические записи сохраняют `object_id`;
- фильтры объекта применяются на backend;
- индексы строятся по `(company_id, object_id, period/status)`.

## Несколько компаний
Каждая доменная таблица содержит `company_id`. Запросы должны фильтроваться по компании до применения пользовательских фильтров.

Правила:
- данные компаний не смешиваются;
- пользователи принадлежат компании;
- объекты принадлежат компании;
- фото storage должен поддерживать разделение по компании;
- аналитика и AI строятся только в доступной области.

## Миллионы фотографий
Фотографии хранятся во внешнем storage. PostgreSQL хранит метаданные.

Масштабирование фото:
- ключ storage уникален;
- индексы по задаче, этапу, смене и автору;
- старые файлы могут перемещаться в холодное хранилище без потери метаданных;
- аналитика использует метаданные, а не обход файлов;
- удаление файла не должно уничтожать историю события без архивного процесса.

## Архивирование
Архивируются:
- компании;
- объекты;
- задачи;
- старые фото;
- исторические уведомления.

Не архивируются как обычные записи без специального процесса:
- монетные транзакции;
- утвержденные выплаты;
- audit log активной истории;
- AI-рекомендации, связанные с финансовыми решениями.

## Резервное копирование
Резервное копирование должно охватывать:
- PostgreSQL;
- файловое хранилище фото;
- соответствие между `photos.storage_key` и файлами;
- audit log;
- финансовые исторические записи.

Резервная копия считается пригодной, если после восстановления можно открыть историю задачи, смены, фото, монет, AI-рекомендации и выплаты.

## Партиционирование
Для больших исторических таблиц допустимо партиционирование по времени и компании:
- `audit_log`;
- `photos`;
- `coin_transactions`;
- `notifications`;
- `ai_recommendations`.

Партиционирование не должно менять бизнес-семантику таблиц.

## Агрегаты
Агрегаты допустимы для финансовых экранов и отчетов:
- сумма монет;
- рабочее время;
- динамика по дням;
- показатели эффективности.

Правило: агрегаты должны быть воспроизводимы из исходных исторических таблиц.

## Миграции
Миграции базы данных должны:
- сохранять исторические финансовые данные;
- не удалять enum-значения без миграции истории;
- добавлять nullable-поля перед обязательными, если данные уже существуют;
- фиксироваться в changelog проекта;
- проходить проверку на связность FK.

## Критерии готовности
База готова к масштабированию, если:
- каждая доменная таблица имеет `company_id`;
- основные FK и индексы описаны;
- фото отделены от структурированных данных;
- исторические записи не теряются;
- финансовые решения восстановимы;
- AI не имеет права напрямую утверждать выплаты;
- резервное восстановление сохраняет связи между сущностями.
