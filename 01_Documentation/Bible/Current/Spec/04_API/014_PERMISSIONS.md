# 014 — Permissions

## Назначение
Документ фиксирует role-based access control для API. UI скрывает недоступные действия, но backend всегда проверяет роль.

## Роли
- `worker` — монтажник.
- `foreman` — прораб.
- `finance` — финансист.
- `manager` — руководительская роль.

## Матрица доступа

| Область | worker | foreman | finance | manager |
|---|---:|---:|---:|---:|
| Auth me | да | да | да | да |
| Свой профиль | да | да | да | да |
| Список сотрудников | ограниченно | да | да | аналитика |
| Объекты | чтение | чтение | чтение | чтение |
| Создание задач | нет | да | нет | нет |
| Взятие задачи | да | нет | нет | нет |
| Выполнение этапа | да | нет | нет | нет |
| Загрузка фото работы | да | да для примеров | чтение | чтение |
| Начало/конец смены | да | нет | нет | нет |
| Просмотр монет | свои | сотрудники | финансы | аналитика |
| AI-рекомендации | нет | нет | да | аналитика |
| Создание выплаты | нет | нет | да | нет |
| История выплат | нет | нет | да | аналитика |
| Уведомления финансиста | нет | нет | да | нет |

## Пример отказа

```json
{
  "error": {
    "code": "FORBIDDEN",
    "message": "Недостаточно прав для создания задачи.",
    "details": {
      "requiredRole": "foreman"
    },
    "requestId": "req_permissions_001"
  }
}
```

## Company scope
Все запросы фильтруются по `companyId` из JWT. Клиент не может передать чужой `companyId` и получить данные другой компании.

## Финансовые ограничения
- Только `finance` создает выплату.
- ИИ не имеет роли утверждающего.
- Повторная выплата за тот же период и объект блокируется.
- Финансовые действия пишутся в `audit_log`.

## Связь с БД
- `users.role`;
- `users.company_id`;
- `audit_log.actor_role`;
- все доменные таблицы с `company_id`.
