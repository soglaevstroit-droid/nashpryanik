# 016 — API Versioning

## Назначение
Документ описывает версионирование API. Текущая версия — `/api/v1`.

## Правило URL

```text
/api/v1/{resource}
```

Примеры:
- `/api/v1/auth/login`;
- `/api/v1/tasks`;
- `/api/v1/photos`;
- `/api/v1/finance/summary`.

## Совместимые изменения
Внутри `/api/v1` допускается:
- добавлять optional-поля в response;
- добавлять новые фильтры без изменения старых;
- добавлять новые enum-значения только после обновления документации;
- расширять `meta`.

## Несовместимые изменения
Требуют новой версии API:
- удаление поля;
- изменение типа поля;
- изменение смысла enum;
- изменение бизнес-логики роли;
- изменение формата ошибок;
- перенос финансового решения с человека на ИИ.

## Deprecation
Если endpoint устаревает, API возвращает заголовок:

```http
Deprecation: true
Sunset: 2027-01-01
```

## Response meta

```json
{
  "data": {},
  "meta": {
    "apiVersion": "v1",
    "requestId": "req_version_001"
  }
}
```

## Правила
- UI Bible и Database Bible должны обновляться до изменения версии.
- `/api/v1` не должен ломать текущие утвержденные экраны.
- Изменение версии не может вводить новую бизнес-логику без Project Bible.
