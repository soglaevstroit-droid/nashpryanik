# 008 — Scalability

## Назначение
Документ описывает принципы масштабирования НАШПРЯНИК.РФ. Масштабирование должно поддерживать рост сотрудников, объектов, компаний, ролей, фото и модулей без изменения утвержденной бизнес-логики.

## Основной принцип
Система должна масштабироваться за счет четких границ модулей, единого источника информации, ролевого доступа, историчности данных и отделения файлового хранения от структурированной бизнес-логики.

## Тысячи сотрудников

### Требования
- Быстрая загрузка списков сотрудников по роли и фильтрам.
- Фильтрация по объекту, статусу и периоду.
- Серверная пагинация или порционная загрузка для длинных списков.
- Индексация по сотруднику, объекту, статусу смены и активной задаче.

### Архитектурные правила
- Главные экраны не должны запрашивать всю историю сотрудника.
- Карточка сотрудника загружает детальные данные только при открытии.
- Аналитика использует агрегаты, а не полный перебор событий на клиенте.
- Ролевые ограничения применяются до выдачи данных.

## Десятки объектов

### Требования
- Все задачи, смены, монеты, выплаты и отчеты должны иметь связь с объектом, если объект применим к сценарию.
- Фильтры объекта должны работать в аналитике, Ганте, выплатах, истории и отчетах.
- Исторические данные объекта сохраняются даже при архивировании.

### Архитектурные правила
- Объект является измерением аналитики и финансовых отчетов.
- Удаление или архивирование объекта не должно разрушать историю.
- Списки задач и сотрудников должны загружаться в контексте объекта.

## Несколько компаний

### Требования
Проект должен допускать расширение на несколько компаний без смешивания данных.

### Архитектурные правила
- Каждая доменная сущность должна быть готова к привязке к компании или рабочему пространству.
- Пользователь должен видеть только данные своей доступной области.
- Аналитика и отчеты не должны смешивать компании.
- Файловое хранилище должно поддерживать разделение по области владения.

## Множество ролей

### Требования
Система уже фиксирует роли монтажника, прораба, финансиста и руководителя как управленческую роль. Расширение ролей допускается только через документацию и не должно обходить текущие ограничения.

### Архитектурные правила
- Проверка роли выполняется на backend.
- UI скрывает недоступные действия, но не является единственной защитой.
- Новая роль должна иметь описанные обязанности, права, ограничения и доступные экраны.
- Нельзя давать ИИ роль утверждающего.

## Большие объемы фото

### Требования
- Фото должно храниться в файловом хранилище.
- База данных хранит метаданные и связи.
- Загрузка должна поддерживать состояния Uploading photo, Error и повтор.
- История должна сохранять связь фото с событием.

### Архитектурные правила
- Нельзя хранить бизнес-смысл только в имени файла.
- Фото связано с автором, временем, задачей, этапом или сменой.
- Аналитика использует метаданные, а не прямой обход файлов.
- Массовое хранение фото не должно замедлять списки задач и сотрудников.

## Расширение модулей

### Требования
Новые модули должны подключаться к существующим событиям, ролям и данным без дублирования бизнес-логики.

### Архитектурные правила
- Модуль не должен создавать второй источник правды.
- Модуль должен иметь понятные входные и выходные данные.
- Модуль должен сохранять историю значимых действий.
- Модуль должен использовать общие состояния ошибок и загрузки.
- Модуль должен соблюдать Design System и UI Bible.

## Масштабирование аналитики

### Требования
Аналитика должна поддерживать периоды, сотрудников, объекты, задачи, этапы, монеты, выплаты и AI-рекомендации.

### Архитектурные правила
- Для часто используемых экранов допустимы агрегаты.
- Агрегаты должны быть воспроизводимы из исторических событий.
- Утвержденные выплаты не пересчитываются задним числом автоматически.
- Неполные данные показываются как неполные, а не как точные.

## Масштабирование AI Engine

### Требования
AI Engine должен работать с растущим объемом задач, фото-метаданных, монет, смен и выплат.

### Архитектурные правила
- AI получает только необходимые подтвержденные данные.
- AI-рекомендация должна иметь объяснение.
- AI не изменяет монеты и выплаты напрямую.
- AI-сигналы могут создавать уведомления, но не финальные решения.
- Повторный расчет AI не должен менять утвержденную историю выплат.

## Масштабирование уведомлений

### Требования
При росте сотрудников и объектов увеличится количество событий, требующих внимания.

### Архитектурные правила
- Уведомления должны иметь тип, статус, приоритет и связанную сущность.
- Списки уведомлений фильтруются по периоду, объекту, типу и статусу.
- Уведомление не выполняет действие автоматически.
- Обработка уведомления сохраняется как событие.

## Масштабирование API

### Требования
API должен выдерживать рост экранов, ролей и данных.

### Архитектурные правила
- Endpoint должен быть привязан к роли и бизнес-действию.
- Ответы экранов должны быть достаточными, но не перегруженными.
- Фильтры периода, объекта и сотрудника применяются на backend.
- Ошибки должны быть типизированы и объяснимы.
- Финансовые endpoint не должны принимать неявные решения.

## Масштабирование базы данных

### Требования
База данных должна поддерживать быстрый доступ к активным сценариям и историческим данным.

### Архитектурные правила
- Индексы нужны по `employeeId`, `objectId`, `taskId`, `period`, `status`, `createdAt`.
- История действий хранится отдельно от текущих представлений.
- Связи между задачами, этапами, фото, монетами и выплатами обязательны.
- Миграции не должны терять исторические финансовые данные.

## Масштабирование интерфейса

### Требования
Интерфейс должен оставаться простым при росте данных.

### Архитектурные правила
- Монтажнику показывается только то, что помогает работать.
- Прорабу показываются сотрудники, статусы и текущая работа.
- Финансисту показываются цифры, факты, история, AI-оценка и премии.
- Большие списки требуют фильтров и порционной загрузки.
- Дизайн сохраняет утвержденный стиль минимум на 95%.

## Критерии готовности к росту
Система готова к масштабированию, если:
- ролевые права проверяются backend;
- ключевые данные имеют связи и историю;
- фото вынесены в файловое хранилище;
- аналитика строится из подтвержденных событий;
- AI не принимает финальные решения;
- выплаты сохраняются как исторические записи;
- интерфейс не перегружается при росте данных.
