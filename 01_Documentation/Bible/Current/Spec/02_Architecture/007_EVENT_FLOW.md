# 007 — Event Flow

## Назначение
Документ описывает события системы: что происходит после пользовательского действия, как событие проходит через UI, API, backend, database, storage, analytics, AI, notifications и возвращается в UI.

## Общий шаблон события

```text
User action
  ↓
UI state update
  ↓
API request
  ↓
Backend validation
  ↓
Database transaction
  ↓
Storage operation, if file exists
  ↓
Domain event
  ↓
Analytics update
  ↓
AI or Notification, if needed
  ↓
API response
  ↓
UI update
```

## Событие: пользователь нажал кнопку
1. UI проверяет локальную доступность действия.
2. UI переводит компонент в состояние Saving или Disabled action.
3. API получает команду.
4. Backend проверяет авторизацию, роль и бизнес-правило.
5. Backend сохраняет событие или возвращает ошибку.
6. UI показывает результат.

Локальная доступность в UI не заменяет backend-проверку.

## Событие: начало рабочей смены
1. Монтажник нажимает начало работы.
2. UI открывает подтверждение.
3. Если требуется фото, запускается камера.
4. Фото отправляется на backend.
5. Backend проверяет роль и состояние смены.
6. Database создает смену и событие `work_started`.
7. UI показывает статус `На работе`.

## Событие: завершение рабочей смены
1. Монтажник нажимает завершение работы.
2. UI открывает подтверждение.
3. Backend проверяет активную смену.
4. Database сохраняет событие `work_finished`.
5. Смена закрывается.
6. UI показывает статус `Ушел`.

## Событие: задача взята в работу
1. Монтажник открывает задачу.
2. UI показывает подтверждение взятия.
3. Backend проверяет, что задача доступна и у сотрудника нет другой задачи в работе.
4. Database сохраняет событие `task_taken`.
5. Задача переходит в состояние `В работе`.
6. UI приглушает остальные задачи.

## Событие: этап завершен
1. Монтажник выполняет этап.
2. UI требует фото, если действие ключевое.
3. API получает фото и команду завершения.
4. Backend проверяет задачу, этап, роль и смену.
5. Storage сохраняет фото.
6. Database сохраняет фото-метаданные.
7. Database сохраняет событие `step_completed`.
8. Этап переходит в состояние `Завершен`.
9. Монеты рассчитываются, если сотрудник `На работе`.
10. Analytics получает событие.
11. UI показывает завершенный этап и обновленный баланс.

## Событие: задача не может быть выполнена
1. Монтажник выбирает `Невозможно выполнить`.
2. UI собирает причину и фото, если требуется.
3. Backend сохраняет событие и связанные данные.
4. Задача или этап получают состояние, отражающее проблему.
5. Прораб получает проверяемый контекст для дальнейшего решения.

## Событие: задача создана прорабом
1. Прораб вводит задачу, исполнителей, срок и этапы.
2. UI отправляет данные.
3. Backend проверяет роль прораба.
4. Database создает задачу и этапы.
5. Система создает событие задачи.
6. Задача появляется в списке доступных сотрудников согласно правилам.

## Событие: начислены монеты
1. Подтвержденное рабочее событие сохранено.
2. Backend проверяет статус `На работе`.
3. Backend создает начисление монет.
4. Database сохраняет начисление с источником.
5. Analytics обновляет агрегаты.
6. UI показывает обновленный баланс.

Монеты не начисляются при статусе `Ушел`.

## Событие: AI-рекомендация создана
1. Analytics или finance запрашивает оценку.
2. Backend собирает подтвержденные данные.
3. AI Engine анализирует данные.
4. AI Engine возвращает рекомендацию и объяснение.
5. Backend сохраняет рекомендацию.
6. Notifications может создать уведомление.
7. UI показывает рекомендацию финансисту.

ИИ не утверждает выплату и не меняет монеты.

## Событие: выплата утверждена
1. Финансист открывает черновик выплаты.
2. Финансист вводит премию и комментарий.
3. UI отправляет решение.
4. Backend проверяет роль, суммы, период и отсутствие повторной выплаты.
5. Database сохраняет выплату и историю решения.
6. Analytics обновляет отчеты.
7. Notification может отметить готовую выплату обработанной.
8. UI показывает сохраненное решение.

## Событие: уведомление просмотрено
1. Пользователь открывает уведомление.
2. UI отправляет событие просмотра.
3. Backend проверяет доступ к уведомлению.
4. Database меняет статус на `Просмотрено`.
5. UI обновляет счетчики.

## Событие: отчет выгружен
1. Финансист выбирает фильтры.
2. UI запрашивает выгрузку.
3. Backend формирует отчет по текущим данным.
4. Файл отдается пользователю.
5. Статусы выплат не меняются.

## Ошибки событий
Каждое событие может завершиться ошибкой:
- нет прав;
- некорректное состояние;
- ошибка валидации;
- ошибка загрузки фото;
- конфликт повторной выплаты;
- ошибка backend;
- потеря сети.

Ошибка должна быть объяснима пользователю и не должна переводить сущность в финальное состояние без сохраненного события.
