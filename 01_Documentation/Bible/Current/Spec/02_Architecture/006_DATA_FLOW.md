# 006 — Data Flow

## Назначение
Документ описывает движение данных между UI, API, backend, database, storage, analytics, AI и finance. Потоки данных должны сохранять проверяемость, ролевую безопасность и единый источник информации.

## Общий поток данных

```text
Mobile UI
  ↓ request
API
  ↓ validation
Backend
  ↓ write/read
Database
  ↓ files
Storage
  ↓ aggregates
Analytics
  ↓ context
AI Engine
  ↓ recommendation
Finance / Notifications
  ↓ response
Mobile UI
```

## Поток авторизации
1. Пользователь вводит данные входа.
2. UI отправляет запрос авторизации.
3. API передает запрос backend.
4. Backend проверяет пользователя.
5. Backend возвращает роль и доступный стартовый экран.
6. UI открывает экран роли.

Критическое правило: роль проверяется не только в UI, но и на backend для каждого защищенного действия.

## Поток рабочей смены
1. Монтажник нажимает начало работы.
2. UI отправляет команду и фото, если оно требуется.
3. API принимает данные.
4. Backend проверяет роль монтажника и отсутствие конфликтов состояния.
5. Database сохраняет смену и событие.
6. UI получает статус `На работе`.
7. Дальнейшие подтвержденные действия могут создавать начисления монет.

При завершении смены поток обратный по смыслу: backend закрывает активную смену, сохраняет событие и возвращает статус `Ушел`.

## Поток задачи
1. Прораб создает задачу.
2. UI отправляет название, описание, объект, исполнителей, срок и этапы.
3. Backend валидирует роль прораба и данные задачи.
4. Database сохраняет задачу и этапы.
5. Задача становится доступной монтажникам согласно правилам назначения.
6. Монтажник открывает список задач.
7. API возвращает задачи с учетом роли, объекта, статуса и активной задачи сотрудника.

## Поток этапа и фото
1. Монтажник выполняет этап.
2. UI запрашивает фото, если действие ключевое.
3. Фото отправляется через API.
4. Backend проверяет:
   - авторизованного пользователя;
   - роль монтажника;
   - активную смену;
   - задачу в работе;
   - доступность этапа.
5. Файл сохраняется в Storage.
6. Database сохраняет метаданные фото.
7. Database сохраняет событие этапа.
8. Backend обновляет статус этапа.
9. Backend запускает расчет монет, если действие подтверждено и сотрудник `На работе`.
10. UI получает обновленные данные задачи.

## Поток монет
1. Событие работы подтверждено.
2. Backend проверяет статус смены `На работе`.
3. Backend применяет утвержденную логику начисления.
4. Database сохраняет начисление как историческое событие.
5. Analytics пересчитывает агрегаты.
6. UI показывает обновленный баланс.
7. Финансовые экраны используют те же начисления.

Монеты не редактируются вручную прорабом или финансистом.

## Поток аналитики
1. Пользователь открывает аналитический экран.
2. UI передает фильтры: период, объект, сотрудник.
3. API отправляет запрос backend.
4. Backend читает подтвержденные данные.
5. Analytics строит агрегаты.
6. Backend возвращает карточки, диаграммы, списки и состояния.
7. UI отображает данные с сохранением Design System.

Если данных недостаточно, возвращается состояние неполных данных, а не точность, которой нет.

## Поток AI-рекомендаций
1. Backend собирает подтвержденные данные: задачи, этапы, смены, монеты, историю активности, объект и период.
2. AI Engine получает данные для анализа.
3. AI Engine формирует рекомендацию и объяснение.
4. Backend сохраняет рекомендацию или возвращает ее экрану.
5. UI показывает рекомендацию отдельно от финального решения.
6. Финансист принимает, отклоняет или откладывает решение в рамках процесса.

ИИ не утверждает выплату и не меняет монеты.

## Поток выплаты
1. Финансист открывает экран выплат.
2. Backend формирует черновик: начисленные монеты, AI-рекомендация, премия, комментарий, итог.
3. Финансист вводит премию и комментарий.
4. UI отправляет решение.
5. Backend проверяет:
   - роль финансиста;
   - период;
   - суммы;
   - отсутствие повторной выплаты;
   - корректность итоговой суммы.
6. Database сохраняет выплату и историю решения.
7. Analytics и отчеты используют утвержденную запись.
8. UI показывает сохраненный результат.

## Поток уведомлений
1. Событие системы, аналитики или AI требует внимания.
2. Backend создает уведомление.
3. Database сохраняет тип, статус, приоритет и связанную сущность.
4. UI получает список уведомлений.
5. Пользователь открывает уведомление.
6. Backend отмечает его просмотренным или обработанным.

Уведомление не выполняет финансовое действие автоматически.

## Поток отчетов
1. Финансист выбирает период, объект и сотрудников.
2. Backend собирает начисленные монеты, премии, итоги и AI-рекомендации.
3. Analytics формирует отчетные строки и итоги.
4. UI показывает отчет.
5. При выгрузке backend формирует файл текущего отчета.
6. Файл отдается пользователю без изменения статусов выплат.

## Требования к данным
- Все финансовые значения должны быть неотрицательными.
- Итог выплаты равен начисленным монетам плюс премия.
- Исторические выплаты не пересчитываются автоматически.
- Фото связано с событием и автором.
- Роль проверяется на backend.
- Все значимые действия попадают в историю.
