# 005 — App Lifecycle

## Назначение
Документ описывает жизненный цикл приложения от запуска до завершения рабочей смены и связанных финансовых процессов. Описание основано на утвержденных ролях, UI Bible и бизнес-правилах.

## 1. Запуск приложения
При запуске приложение подготавливает интерфейс, проверяет наличие сессии и состояние сети.

Система должна:
- показать безопасное стартовое состояние;
- проверить авторизацию;
- определить роль пользователя;
- загрузить минимальный набор данных для стартового экрана;
- обработать Offline или Error без поломки композиции.

## 2. Авторизация
Пользователь проходит через `000_Login`. После успешного входа backend возвращает роль и разрешенный контекст.

Стартовый экран зависит от роли:
- монтажник → `001_Worker_Main`;
- прораб → `015_Foreman_Employees`;
- финансист → `029_Finance_Main`.

Руководительская аналитика в MVP не имеет отдельного набора экранов и должна использовать проверяемые данные без обхода ролевых правил.

## 3. Инициализация роли

### Монтажник
Система загружает:
- имя сотрудника;
- статус `На работе` или `Ушел`;
- баланс монет;
- таймер смены;
- общий список задач объекта;
- доступность задач.

### Прораб
Система загружает:
- список сотрудников;
- статусы;
- текущие задачи;
- прогресс этапов;
- монеты и время;
- сообщения и черновики, если они нужны экрану.

### Финансист
Система загружает:
- общую сумму монет;
- количество работающих сотрудников;
- список сотрудников;
- финансовую аналитику;
- AI-рекомендации и уведомления, если они нужны экрану.

## 4. Рабочий цикл монтажника

### Начало смены
1. Монтажник нажимает начало работы.
2. UI открывает подтверждение.
3. Если требуется фото, открывается камера.
4. Фото отправляется через API.
5. Backend проверяет пользователя, роль и корректность действия.
6. Смена переводится в статус `На работе`.
7. UI обновляет таймер и доступность действий.

### Выбор задачи
1. Монтажник открывает общий список задач объекта.
2. UI показывает доступность задач.
3. Монтажник открывает задачу.
4. Если задача свободна или назначена ему, он может взять ее.
5. Backend проверяет, что у сотрудника нет другой задачи в работе.
6. Задача переводится в состояние `В работе`.
7. Остальные задачи становятся приглушенными и доступны только для просмотра.

### Выполнение этапа
1. Монтажник открывает этап.
2. UI показывает описание и фото-пример, если он есть.
3. Монтажник выполняет действие.
4. Система требует фото, если действие ключевое.
5. Фото загружается.
6. Backend сохраняет событие, фото и состояние этапа.
7. Если сотрудник `На работе`, монеты начисляются по утвержденной логике.
8. UI обновляет прогресс, историю и баланс.

### Невозможно выполнить
1. Монтажник выбирает действие невозможности выполнения.
2. Система фиксирует причину и фото, если это требуется.
3. Backend сохраняет событие.
4. Прораб получает проверяемую информацию о проблеме через рабочие экраны или уведомления.

### Завершение смены
1. Монтажник нажимает завершение работы.
2. UI открывает подтверждение.
3. Backend закрывает активную смену.
4. Сотрудник получает статус `Ушел`.
5. Новые монеты не начисляются до следующего начала работы.

## 5. Рабочий цикл прораба
1. Прораб открывает список сотрудников.
2. Проверяет статусы, текущие задачи, прогресс, монеты и время.
3. Создает задачу при необходимости.
4. Добавляет исполнителей, срок и этапы.
5. При необходимости добавляет фото-примеры к этапам.
6. Сохраняет задачу.
7. Задача появляется у доступных сотрудников согласно правилам назначения.
8. Прораб контролирует выполнение и общается через сообщения.

Прораб не начисляет монеты вручную и не принимает финансовые решения по выплатам.

## 6. Финансовый цикл
1. Финансист открывает главную финансовую картину.
2. Система показывает общие монеты, сотрудников и аналитику.
3. Финансист открывает сотрудника, Гант, отчеты или AI-рекомендации.
4. AI Engine может сформировать рекомендацию на основании подтвержденных данных.
5. Финансист открывает экран выплат.
6. UI показывает начисленные монеты, AI-рекомендацию, премию, комментарий и итог.
7. Финансист принимает решение.
8. Backend проверяет роль, суммы, период и отсутствие повторной выплаты.
9. Выплата сохраняется в истории.
10. Отчеты и аналитика используют сохраненную историю.

## 7. Системные состояния
На всех этапах приложение должно корректно обрабатывать:
- Loading;
- Empty;
- Ready;
- Error;
- Offline;
- Saving;
- Uploading photo;
- Disabled action;
- Conflict.

Состояния не должны ломать композицию экранов и не должны скрывать бизнес-ограничения.

## 8. Завершение пользовательской сессии
Сессия может завершиться выходом пользователя или истечением авторизации. После этого защищенные экраны и API-действия недоступны до повторного входа.

Если пользователь выходит во время незавершенного действия, backend не должен считать действие завершенным без сохраненного события и подтверждения.
