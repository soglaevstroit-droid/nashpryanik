# 001 — System Architecture

## Назначение
Документ описывает общую архитектуру НАШПРЯНИК.РФ. Система строится как связанная платформа, где пользовательские действия проходят через UI, API, backend, базу данных, файловое хранилище, аналитику, AI Engine и уведомления.

## Архитектурный принцип
Единый источник информации важнее локальных экранных решений. Любой показатель, видимый в UI Bible, финансовой аналитике, отчетах или AI-рекомендациях, должен происходить из согласованной бизнес-логики и проверяемых данных.

## Компоненты системы

### Мобильное приложение
Мобильный интерфейс является основным рабочим слоем для пользователей. Все экраны проектируются сначала под iPhone и сохраняют утвержденный стиль референсов минимум на 95%.

Мобильное приложение отвечает за:
- авторизацию пользователя;
- показ экранов согласно роли;
- ввод действий пользователя;
- загрузку фото;
- отображение состояний Loading, Empty, Ready, Error, Offline, Saving, Uploading photo и Disabled action;
- получение данных через API;
- передачу команд backend.

### Backend
Backend является центральным слоем бизнес-логики. Он проверяет роли, валидирует действия, управляет жизненным циклом задач, смен, этапов, фото, монет, AI-рекомендаций и выплат.

Backend отвечает за:
- проверку авторизации и роли;
- применение бизнес-правил;
- обработку API-запросов;
- запись событий;
- работу с базой данных;
- передачу фото в файловое хранилище;
- запуск аналитических и AI-процессов;
- создание уведомлений.

### API
API является контрактом между UI и backend. Каждый защищенный endpoint обязан проверять роль пользователя на сервере, даже если действие скрыто в интерфейсе.

API отвечает за:
- получение экранных данных;
- создание задач, этапов, смен и выплат;
- загрузку фото через безопасный процесс;
- выдачу аналитики и отчетов;
- возврат ошибок в понятной форме;
- сохранение истории значимых действий.

### База данных
База данных хранит структурированные данные системы:
- пользователей и роли;
- сотрудников;
- объекты;
- задачи;
- этапы;
- рабочие смены;
- фото-метаданные;
- начисленные монеты;
- AI-рекомендации;
- уведомления;
- выплаты;
- историю действий;
- настройки финансиста.

Исторические записи не должны терять связь с сотрудником, объектом, задачей, этапом, сменой и финансовым решением.

### Файловое хранилище
Файловое хранилище хранит фото, которые подтверждают ключевые действия. База данных хранит метаданные и ссылки на файлы, а не должна полагаться только на неструктурированные имена файлов.

Файловое хранилище отвечает за:
- прием фото;
- хранение оригинала или рабочей версии;
- выдачу безопасной ссылки;
- поддержку истории и аудита;
- масштабирование больших объемов фото.

### AI Engine
AI Engine анализирует подтвержденные данные и формирует рекомендации. Он не принимает финальные решения и не изменяет бизнес-данные напрямую.

AI Engine может:
- сравнивать начисленные монеты со своей оценкой стоимости работы;
- рекомендовать премию;
- объяснять отклонения;
- подсвечивать подозрительные действия;
- формировать сигналы для уведомлений.

AI Engine не может:
- утверждать выплату;
- начислять монеты;
- менять статус задачи;
- скрывать событие;
- принимать финальное управленческое решение.

### Уведомления
Система уведомлений сообщает пользователям о событиях, требующих внимания. Уведомление не выполняет действие само по себе.

Уведомления используются для:
- новых AI-рекомендаций;
- больших отклонений;
- сотрудников без активности;
- подозрительных действий;
- готовых выплат;
- ошибок или событий, требующих проверки.

### Аналитика
Аналитика строится на подтвержденных данных: задачах, этапах, сменах, фото, монетах, выплатах, истории и AI-рекомендациях.

Аналитика используется для:
- общей суммы монет;
- динамики по дням, объектам и сотрудникам;
- диаграммы Ганта;
- эффективности;
- финансовых отчетов;
- истории выплат;
- KPI проекта.

### Роли пользователей
Архитектура поддерживает роли, утвержденные в `004_USER_ROLES.md`:
- монтажник;
- прораб;
- финансист;
- руководитель как управленческая роль без отдельного набора MVP-экранов.

Роли определяют доступ к экранам, API, данным и действиям. Ролевые ограничения должны проверяться на UI и backend.

## Взаимодействие компонентов

```text
User
  ↓
Mobile UI
  ↓
API
  ↓
Backend Business Logic
  ↓
Database ←→ Analytics
  ↓              ↓
File Storage     AI Engine
  ↓              ↓
Notifications ← Events
  ↓
Mobile UI Update
```

## Пример: выполнение этапа монтажником
1. Монтажник открывает текущую задачу.
2. UI показывает этап и требует фото, если действие ключевое.
3. Пользователь делает фото.
4. UI отправляет фото и команду через API.
5. Backend проверяет роль, активную смену, текущую задачу и правило одной задачи.
6. Фото сохраняется в файловое хранилище.
7. Метаданные фото и событие этапа сохраняются в базе данных.
8. Если действие подтверждено и сотрудник `На работе`, backend применяет правило начисления монет.
9. Аналитика получает новое событие.
10. UI обновляет состояние задачи и баланса монет.

## Пример: рекомендация и выплата
1. Финансист открывает AI-рекомендации.
2. Backend собирает подтвержденные данные по сотруднику, задачам, времени и монетам.
3. AI Engine формирует рекомендацию и объяснение.
4. UI показывает рекомендацию отдельно от решения.
5. Финансист указывает премию и комментарий.
6. Backend проверяет суммы, период, роль и отсутствие повторной выплаты.
7. Выплата сохраняется в истории.
8. Аналитика и отчеты используют утвержденную запись.

## Архитектурные границы
- UI не должен подменять backend-валидацию.
- Backend не должен принимать финансовое действие без явного решения финансиста.
- AI Engine не должен напрямую изменять выплаты и монеты.
- File Storage не является источником бизнес-логики.
- Analytics не должна пересчитывать утвержденную историю выплат задним числом.
