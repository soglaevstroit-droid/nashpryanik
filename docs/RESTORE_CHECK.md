# Изолированная проверка восстановления

```bash
npm run backup:restore-check
npm run backup:restore-check -- --latest --json
npm run backup:restore-check -- --file stroit-20260712-030001.sql.gz
```

Команда сначала требует успешный `backup:check`, принимает только basename из
`/root/backups/postgres`, запрещает ссылки, пустые файлы и неожиданные имена. Формат определяется
по расширению и проверяется соответствующим инструментом: `gzip -t` для `.sql.gz` или
`pg_restore --list` для custom dump.

Временная база имеет обязательный префикс `stroit_restore_check_`. До `CREATE DATABASE`
проверяется отсутствие базы и запрет имени `stroit_dev`. Восстановление никогда не направляется
в production. После него выполняются только агрегированные `SELECT`: таблицы, миграции,
пользователи и задачи. Пароли и персональные данные не выводятся.

Очистка работает через `trap` и локальный `try/finally`: при успехе, ошибке, timeout, SIGINT и
SIGTERM завершаются подключения только к временной базе, после чего удаляется только имя с
безопасным префиксом. Активное имя дополнительно хранится в
`.runtime/restore-check-active.json`. Если очистка не подтверждена, команда завершится CRITICAL
и напечатает точное имя для ручной диагностики. Никогда не копируйте команду удаления, не
проверив префикс и запрет `stroit_dev`.
